# Architecture Issue: Duplicate Streaming Implementation

## Problem

Two nearly identical streaming handlers in the same file:

### Handler 1: `handleSubmit()` (lines 170-276)
```typescript
const handleSubmit = async () => {
  // ... setup code
  for await (const chunk of streamChatCompletion(apiMessages, config, true, signal)) {
    if (chunk.content) {
      const contentStr = String(chunk.content)
      if (contentStr.includes('[object Object]') || contentStr.trim().startsWith('{')) {
        console.log('Skipping object-like content:', ...)
      } else {
        assistantContent += contentStr
        assistantMessage.content = assistantContent
        setMessages([...newMessages, assistantMessage])
      }
    }
    
    if (chunk.toolCalls) {
      // Accumulate tool calls
      for (const tc of chunk.toolCalls) {
        const idx = tc.index ?? 0
        const existing = pendingToolCallsMap.get(idx)
        if (existing) {
          existing.function.name = tc.function?.name || existing.function?.name
          existing.function.arguments += tc.function?.arguments || ''
        } else {
          pendingToolCallsMap.set(idx, {...})
        }
      }
    }
  }
  
  // Execute pending tool calls
  const pendingToolCalls = Array.from(pendingToolCallsMap.values())
  if (pendingToolCalls.length > 0) {
    for (const toolCall of pendingToolCalls) {
      if (toolCall.function.name === 'open_terminal') {...}
      else if (toolCall.function.name === 'execute_command') {...}
    }
  }
}
```

## Differences (Minor)

| Aspect | handleSubmit | generateResponse |
|--------|-------------|------------------|
| Input | user message | any message array |
| Tools handled | all tools | only `execute_command` |
| Called from | user submit | step retry, completion |
| Message array | `newMessages` | `currentMessages` |

## Why This Is Bad

1. **Bug duplication** - Fix streaming bug in one place, miss the other
2. **Inconsistent behavior** - Tool support differs between paths
3. **Maintenance overhead** - 200 lines of nearly identical code
4. **Cognitive load** - Developer must understand why two exist

## Solution

Single streaming hook used by both:

```typescript
// src/hooks/useStreaming.ts
export function useStreaming() {
  const executeStreaming = async ({
    messages,
    onContent,
    onToolCall,
    enabledTools = ['execute_command', 'open_terminal', ...]
  }: StreamingOptions) => {
    for await (const chunk of streamChatCompletion(messages, config, true, signal)) {
      if (chunk.content) onContent(chunk.content)
      if (chunk.toolCalls) {
        const calls = accumulateToolCalls(chunk.toolCalls)
        for (const tool of calls) {
          if (enabledTools.includes(tool.function.name)) {
            await onToolCall(tool)
          }
        }
      }
    }
  }
  
  return { executeStreaming }
}

// Usage in page.tsx:
const { executeStreaming } = useStreaming()

// handleSubmit:
await executeStreaming({messages, onContent, onToolCall, enabledTools: ALL_TOOLS})

// generateResponse:
await executeStreaming({messages, onContent, onToolCall, enabledTools: ['execute_command']})
```

## Alternative

Eliminate `generateResponse` entirely - it's only used for step retries. Handle that through normal message flow:

```typescript
// Instead of:
generateResponse([...messages, retryMsg])

// Just add message and let normal flow handle it:
setMessages([...messages, retryMsg])
// (Effect or callback triggers streaming)
```
