# Architecture Issue: Debug Code Pollution

## Problem

Debug infrastructure is mixed with production code in `tui-debug/` folder and imports:

### Files in tui-debug/
```
src/tui-debug/
  cli.tsx                      # Debug CLI tool
  components/
    StepChainDebugger.tsx      # Execution chain visualizer
    StorageViewer.tsx          # State inspection
    ChatPanel.tsx              # Debug chat interface
  hooks/
    useExecutionState.ts       # Debug tracking hook
    useApiDebug.ts             # API debugging
    useChatStorage.ts          # Chat storage debugging
    useLlmChat.ts              # LLM chat debugging
  utils/
    browser-adapters.ts        # Debug adapters
    index.ts                   # Debug utilities
  package.json               # Separate dependencies
  tsconfig.json              # Separate config
```

### Production Imports Debug Code

```typescript
// page.tsx (before recent changes)
import { useExecutionState } from '@/tui-debug/hooks/useExecutionState'

// Lines 126-127 (now removed)
const [execState, execActions] = useExecutionState()
const [showExecDebugger, setShowExecDebugger] = useState(false)
```

### Debug State Intertwined

```typescript
// Debug tracking mixed with business logic
execActions.recordPlanStart(`plan-${Date.now()}`, args.goal, steps)
execActions.saveCheckpoint()
execActions.recordStepModify(stepIndex, args.newCommand)
execActions.recordContinuationTrigger(currentStepIndex, nextStepIndex, 'success')
execActions.recordTimeoutRegister(timeoutId, description)
execActions.recordGenerateResponse(stepIndex)
```

## Why This Is Bad

1. **Production bloat** - Debug code shipped to users
2. **Performance impact** - Debug tracking adds overhead
3. **Code clutter** - Business logic mixed with instrumentation
4. **Build complexity** - Separate tsconfig, package.json
5. **Testing confusion** - Tests must handle debug state

## The Execution Debugger

The `useExecutionState` hook tracked:
- Plan starts/completions
- Step execution chain
- Timeout registrations
- Retry attempts
- State checkpoints

All for debugging step chain health - but this belongs in development tools, not production code.

## Solution

### Option 1: Conditional Compilation (Preferred)

```typescript
// Use environment variable to strip debug code at build
const DEBUG = process.env.NEXT_PUBLIC_DEBUG === 'true'

function useChat() {
  const [messages, setMessages] = useState([])
  
  // Debug instrumentation only in dev
  const debug = DEBUG ? useDebugTracking() : null
  
  const sendMessage = async (msg) => {
    debug?.recordAction('send', msg)
    // ... normal logic
  }
}
```

### Option 2: Separate DevTools Panel

```typescript
// Debug UI only loads in development
{process.env.NODE_ENV === 'development' && (
  <DevToolsPanel>
    <StepChainDebugger />
    <StorageViewer />
  </DevToolsPanel>
)}
```

### Option 3: Extension-Based Debugging

```typescript
// Use Redux DevTools or similar extension
// No code changes to production
import { devtools } from 'zustand/middleware'

const useStore = create(devtools(store, { name: 'Rogius' }))
```

## Recommended Structure

```
src/
  debug/                      # Debug-only code
    components/               # DevTools UI
    hooks/                   # Debug instrumentation
    utils/                   # Debug utilities
  app/
    page.tsx                 # No debug imports
  
// debug/ folder excluded from production build
```

## Cleanup Required

Based on your recent changes, you've already removed:
- `useExecutionState` import and usage
- `execActions` calls
- `showExecDebugger` state

Consider also removing:
- `tui-debug/` folder imports from production
- Debug logging (`console.log` in streaming handlers)
