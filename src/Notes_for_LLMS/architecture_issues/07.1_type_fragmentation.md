# Architecture Issue: Type Definition Fragmentation

## Problem

Same concepts defined in multiple files with slight variations:

### ChatMessage - Defined 3 Times

```typescript
// src/lib/api-client.ts lines 1-4
export interface ChatMessage {
  role: 'system' | 'user' | 'assistant'
  content: string
}

// src/lib/chat-storage.ts lines 1-8
import { ChatMessage } from './api-client'
export interface BranchedMessage extends ChatMessage {
  messageId: string
  branches?: Branch[]
  currentBranchIndex?: number
}

// src/lib/python-bridge.ts lines 62-65
export interface AIChatMessage {
  role: 'system' | 'user' | 'assistant';
  content: string;
}
```

**Note**: `AIChatMessage` is identical to `ChatMessage` but redefined!

### ToolCall - Defined 2 Times

```typescript
// src/lib/api-client.ts lines 270-278
export interface ToolCall {
  id?: string
  index?: number
  type: 'function'
  function: {
    name?: string
    arguments: string
  }
}

// src/lib/python-bridge.ts lines 75-85
export interface AIChatResponse {
  tool_calls?: Array<{
    id?: string;
    index: number;
    type: string;
    function: {
      name: string;
      arguments: string;
    };
  }>;
}
```

**Differences**:
- `ToolCall` vs `tool_calls` (naming)
- `type: 'function'` vs `type: string` (specificity)
- `name?: string` vs `name: string` (optionality)

### SystemInfo - Defined 2 Times

```typescript
// src/tools/terminal/store.ts lines 23-36
export interface SystemInfo {
  os: string
  kernel: string
  arch: string
  user: string
  home: string
  shell: string
  packageManager: string
  hasSudo: boolean
  pythonVersion: string
  nodeVersion: string
  dockerVersion: string
  collected: boolean

// src/lib/python-bridge.ts lines 100-113
export interface SystemInfoResponse {
  os: string;
  os_version: string;      // Different name!
  architecture: string;      // Different name!
  shell: string;
  hostname: string;        // Extra field!
  username: string;        // Different name!
  python_version: string;  // Snake_case!
  working_directory: string; // Different!
  package_manager: string; // Snake_case!

  node_version: string;    // Snake_case!
  docker_version: string;  // Snake_case!

```

**Mapping required between these!** (`src/tools/terminal/store.ts` lines 107-119)

## Why This Is Bad

1. **No type safety** - Converting between types risks errors
2. **Maintenance burden** - Change field name, update 3 files
3. **Confusion** - Which SystemInfo should I use?
4. **Runtime errors** - TypeScript allows it but data shape differs

## Solution

Centralized types with clear contracts:

```typescript
// src/types/index.ts - Single source of truth

// Base message (API-neutral)
export interface Message {
  role: 'system' | 'user' | 'assistant'
  content: string
}

// Extended for UI features
export interface MessageWithMetadata extends Message {
  id: string
  timestamp: number
  branches?: MessageBranch[]
}

// Tool call (shared)
export interface ToolCall {
  id: string
  index: number
  type: 'function'
  function: {
    name: string
    arguments: string  // JSON string
  }
}

// System info (canonical)
export interface SystemInfo {
  os: string
  osVersion: string      // camelCase consistently
  architecture: string
  shell: string
  hostname: string
  username: string
  pythonVersion: string
  workingDirectory: string
  packageManager: string
  hasSudo: boolean
  nodeVersion: string
  dockerVersion: string
}

// API-specific wrappers (if needed)
export type SystemInfoApiResponse = SnakeCaseKeys<SystemInfo>
// Use utility type to convert camelCase → snake_case for API
```
