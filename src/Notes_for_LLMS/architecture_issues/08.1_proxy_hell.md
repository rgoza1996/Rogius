# Architecture Issue: Proxy Hell (Unnecessary API Indirection)

## Problem

Every Python API call goes through a Next.js proxy route:

### Current Flow
```
Frontend ──▶ Next.js API Route ──▶ Python FastAPI ──▶ Result
           (Node.js server)       (Python backend)
```

### The Proxy Routes

```typescript
// src/app/api/python/terminal/route.ts
export async function POST(request: NextRequest) {
  const body = await request.json()
  const response = await fetch(`${PYTHON_API_URL}/terminal/execute`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body),
  })
  const data = await response.json()
  return NextResponse.json(data)
}
```

This is a **pass-through proxy** - no transformation, no caching, no auth.

### All Proxy Routes

| Route | Proxies To | Purpose |
|-------|-----------|---------|
| `/api/python/ai/chat` | `/ai/chat` | Chat completion |
| `/api/python/multistep/*` | `/multistep/*` | Multi-step tasks |
| `/api/python/settings` | `/settings` | Config management |
| `/api/python/system/info` | `/system/info` | OS detection |
| `/api/python/terminal/*` | `/terminal/*` | Command execution |

**5 proxy routes, all doing the same thing: forward request, return response.**

## Why This Is Bad

1. **Extra hop** - Every request goes through two servers
2. **Latency** - 2 HTTP round trips instead of 1
3. **Complexity** - More code to maintain and debug
4. **Failure points** - 2x the network connections that can fail
5. **CORS confusion** - Proxy "solves" CORS but adds indirection

## The Original Problem (CORS)

Direct fetch from browser to Python backend fails:
```
Browser ──▶ http://127.0.0.1:8000/terminal/execute
            ❌ CORS error - Python doesn't send Access-Control-Allow-Origin
```

## Better Solutions

### Option 1: Enable CORS on Python Backend (Recommended)

```python
# src/tui/api_server.py
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],  # Next.js dev server
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**Result**: Frontend talks directly to Python. Delete all proxy routes.

### Option 2: Use Next.js Rewrites (Simpler)

```javascript
// next.config.js
module.exports = {
  async rewrites() {
    return [
      {
        source: '/api/python/:path*',
        destination: 'http://127.0.0.1:8000/:path*',
      },
    ]
  },
}
```

**Result**: No proxy route files needed. Next.js handles at edge.

### Option 3: tRPC or GraphQL (If Adding Features)

If you need transformation/caching, use proper API layer:
```typescript
// But only if you're ADDING value, not just proxying
```

## Solution: Direct Access + CORS

```typescript
// src/lib/python-bridge.ts
const PYTHON_API_BASE = process.env.NEXT_PUBLIC_PYTHON_API_URL || 'http://127.0.0.1:8000';

export class PythonBridge {
  async terminalExecute(request: TerminalExecuteRequest) {
    // Direct fetch - no proxy!
    const response = await fetch(`${PYTHON_API_BASE}/terminal/execute`, {...})
    return response.json()
  }
}
```

```python
# src/tui/api_server.py - Add CORS
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000", "http://localhost:3001"],
    allow_methods=["*"],
    allow_headers=["*"],
)
```

## Files to Delete

- `src/app/api/python/ai/chat/route.ts`
- `src/app/api/python/multistep/route.ts`
- `src/app/api/python/settings/route.ts`
- `src/app/api/python/system/info/route.ts`
- `src/app/api/python/terminal/route.ts`

**Simpler, faster, less code.**

## Security Considerations

CORS is safe because:
- Python backend runs locally (127.0.0.1)
- Only allows localhost origins
- No production deployment exposure
- Alternative: Proxy is security theater - same network exposure
